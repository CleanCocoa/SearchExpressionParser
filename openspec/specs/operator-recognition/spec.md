## Purpose

The tokenizer recognizes search operators (`AND`, `OR`, `NOT`, `!`) during lexical analysis and distinguishes them from plain words. Operator recognition is case-sensitive, requiring exact ALL-CAPS spelling, and depends on surrounding whitespace to prevent false matches on word prefixes. The extractor priority order determines which token type is produced when multiple extractors could match.

## Requirements

### Requirement: Extractor Priority Order

The system MUST attempt extractors in this fixed order: parens, phrase, bang, NOT, AND, OR, word. The first extractor whose preconditions match and whose extraction succeeds produces the token. The word extractor acts as a fallback and MUST be attempted last.

#### Scenario: Ambiguous input resolved by priority

- **GIVEN** the input `!word`
- **WHEN** the tokenizer processes the `!` character
- **THEN** the bang extractor (higher priority than word) matches first, producing `UnaryOperator.bang` followed by `Word("word")`

### Requirement: AND Binary Operator

The system MUST recognize `AND` as `BinaryOperator.and` only when the three characters `A`, `N`, `D` appear in exact uppercase AND the character at position 3 is whitespace. The extractor consumes 3 characters.

#### Scenario: AND between words

- **GIVEN** the input `you AND me`
- **WHEN** tokenized
- **THEN** the result is `[Word("you"), BinaryOperator.and, Word("me")]`

#### Scenario: AND at end of input (no trailing whitespace)

- **GIVEN** the input `AND`
- **WHEN** tokenized
- **THEN** the result is `[Word("AND")]` because there is no whitespace after the keyword

#### Scenario: AND as word prefix

- **GIVEN** the input `ANDromeda`
- **WHEN** tokenized
- **THEN** the result is `[Word("ANDromeda")]` because position 3 is `r`, not whitespace

#### Scenario: Mixed-case and is a plain word

- **GIVEN** the input `And me` (or `and`, `aNd`, `anD`, `ANd`, `AnD`, `aND`)
- **WHEN** tokenized
- **THEN** the result is two plain words, not an operator

### Requirement: NOT Unary Operator

### Requirement: Bang Unary Operator

The system MUST recognize `!` as `UnaryOperator.bang` only when the character at position 0 is `!` AND the character at position 1 is not whitespace and not nil. The extractor consumes 1 character.

#### Scenario: Bang before a word

- **GIVEN** the input `!word`
- **WHEN** tokenized
- **THEN** the result is `[UnaryOperator.bang, Word("word")]`

#### Scenario: Bang followed by whitespace is a plain word

- **GIVEN** the input `! word`
- **WHEN** tokenized
- **THEN** the result is `[Word("!"), Word("word")]` because position 1 is whitespace

#### Scenario: Lone bang is a plain word

- **GIVEN** the input `!`
- **WHEN** tokenized
- **THEN** the result is `[Word("!")]` because position 1 is nil (end of input)

#### Scenario: Chained bangs

- **GIVEN** the input `!!!!!foo`
- **WHEN** tokenized
- **THEN** the result is `[UnaryOperator.bang, UnaryOperator.bang, UnaryOperator.bang, UnaryOperator.bang, UnaryOperator.bang, Word("foo")]`

#### Scenario: Trailing bang in a word

- **GIVEN** the input `this!`
- **WHEN** tokenized
- **THEN** the result is `[Word("this!")]` because the word extractor consumes `!` as part of the word before bang is tried

### Requirement: Escaping Operators with Backslash

The system MUST treat a backslash as an escape character in word context. When the word extractor encounters `\`, it skips the backslash and includes the next character literally. This allows escaping operator keywords: `\AND` starts with `\`, so the AND extractor precondition (first char `A`) fails, and the word extractor produces `Word("AND")`.

#### Scenario: Escaped AND

- **GIVEN** the input `\AND`
- **WHEN** tokenized
- **THEN** the result is `[Word("AND")]` because `\` prevents the AND extractor from matching

#### Scenario: Escaped backslash

- **GIVEN** the input `\\`
- **WHEN** tokenized
- **THEN** the result is `[Word("\\")]` (a single backslash character)
