## Purpose

SearchExpressionParser is a Swift library that transforms search query strings into evaluatable expression trees. The architecture follows a three-stage pipeline: tokenization converts a raw string into typed tokens via a priority-ordered extractor chain, parsing transforms the token array into a tree of expression nodes using recursive descent, and an optional normalization stage converts the tree to negation normal form for phrase extraction.

## Requirements

### Requirement: Three-Stage Pipeline

The system SHALL process search strings through a three-stage pipeline: Tokenization, Parsing, and Evaluation/Normalization. The `Parser.parse(searchString:)` static method SHALL orchestrate the first two stages as the primary public entry point.

#### Scenario: End-to-end parsing

- **GIVEN** a search string such as `"foo AND bar"`
- **WHEN** `Parser.parse(searchString:)` is called
- **THEN** the Tokenizer SHALL produce a `[Token]` array, the Parser SHALL consume it, and the method SHALL return an `Expression` tree

### Requirement: Tokenization Module

The Tokenizer SHALL convert a raw `String` into an array of `Token` values by iterating a `TokenCharacterBuffer` against a priority-ordered chain of `TokenExtractor` implementations.

#### Scenario: Character buffer consumption

- **GIVEN** a search string input
- **WHEN** the Tokenizer processes it
- **THEN** a `TokenCharacterBuffer` SHALL be created from the string, providing `peekNext`, `consume`, and `resetTo` operations over both original and lowercased character arrays

#### Scenario: Extractor chain ordering

- **GIVEN** the default extractor chain
- **WHEN** token extraction occurs
- **THEN** extractors SHALL be tried in priority order: `OpeningParensExtractor`, `ClosingParensExtractor`, `PhraseExtractor`, `BangExtractor`, `NotExtractor`, `AndExtractor`, `OrExtractor`, `WordExtractor` (wildcard, always last)

#### Scenario: Extractor selection

- **GIVEN** a buffer position and the extractor chain
- **WHEN** the Tokenizer calls `next(buffer:extractors:)`
- **THEN** each extractor's `matchesPreconditions(_:)` SHALL be checked first, and only matching extractors SHALL attempt `extract(_:)`. The first successful extraction SHALL be returned.

#### Scenario: Whitespace handling

- **GIVEN** whitespace characters between tokens
- **WHEN** the Tokenizer advances
- **THEN** whitespace SHALL be skipped before each extraction attempt

### Requirement: TokenExtractor Protocol

All token extractors SHALL conform to the `TokenExtractor` protocol, which defines `matchesPreconditions(_:) -> Bool` and `extract(_:) -> Either<Token, TokenizerError>`.

#### Scenario: Extractor failure

- **GIVEN** an extractor whose preconditions match but extraction fails
- **WHEN** `extract(_:)` returns an `Either.error`
- **THEN** the Tokenizer SHALL collect the error and try the next extractor. If no extractor succeeds, the first error SHALL be returned.

### Requirement: Token Types

The tokenization module SHALL produce the following token types: `Phrase` (aliased as `Word`), `OpeningParens`, `ClosingParens`, `BinaryOperator` (`.and`, `.or`), and `UnaryOperator` (`.bang`, `.not`). All tokens SHALL conform to the `Token` protocol requiring a `string` property.

#### Scenario: Token protocol conformance

- **GIVEN** any token produced by the Tokenizer
- **WHEN** the token is accessed
- **THEN** it SHALL expose a `string` property returning the original text representation

### Requirement: Either Monad for Error Handling

The system SHALL use an `Either<T, E: Error>` enum with `.value(T)` and `.error(E)` cases for internal error propagation during tokenization. This type originates from the DDMathParser heritage.

#### Scenario: Tokenizer result type

- **GIVEN** the Tokenizer's internal operations
- **WHEN** an extractor produces a result
- **THEN** it SHALL be wrapped as `Either<Token, TokenizerError>`, allowing the caller to pattern-match on success or failure without throwing

### Requirement: Parenthesis Balancing

The Parser SHALL balance unmatched parentheses before recursive descent parsing. The `balanceParentheses(tokens:)` function SHALL convert unmatched `OpeningParens` and `ClosingParens` tokens into `Word` tokens containing their literal string representation.

#### Scenario: Unmatched closing parenthesis

- **GIVEN** a token array containing a `ClosingParens` at root level with no matching opener
- **WHEN** `balanceParentheses` runs
- **THEN** the `ClosingParens` SHALL be replaced with a `Word(")")`

#### Scenario: Unmatched opening parenthesis

- **GIVEN** a token array containing an `OpeningParens` with no matching closer
- **WHEN** `balanceParentheses` runs
- **THEN** the `OpeningParens` SHALL be replaced with a `Word("(")`

### Requirement: Recursive Descent Parser

The Parser SHALL use recursive descent to transform a balanced `[Token]` array into an `Expression` tree via a `TokenBuffer` abstraction. The grammar SHALL support primary expressions (negation, parenthesized groups, contains terms) and binary operators (AND, OR) with implicit AND for adjacent terms.

#### Scenario: Implicit AND

- **GIVEN** two adjacent word tokens with no explicit operator
- **WHEN** the Parser encounters a non-operator token after a primary expression
- **THEN** the Parser SHALL produce an `AndNode` combining both expressions

#### Scenario: Empty input

- **GIVEN** an empty token array
- **WHEN** `expression()` is called
- **THEN** the Parser SHALL return an `AnythingNode` wildcard

#### Scenario: Negation parsing

- **GIVEN** a unary operator token (`!` or `NOT`) followed by a primary expression
- **WHEN** `parseNegation` processes the tokens
- **THEN** the Parser SHALL wrap the subsequent primary expression in a `NotNode`

#### Scenario: Parenthesized group

- **GIVEN** matched `OpeningParens` and `ClosingParens` tokens surrounding an expression
- **WHEN** `parseOpeningParens` processes the tokens
- **THEN** the Parser SHALL recursively parse the inner expression as a subgroup

#### Scenario: Empty parentheses

- **GIVEN** an `OpeningParens` immediately followed by a `ClosingParens`
- **WHEN** `parseOpeningParens` processes the tokens
- **THEN** the Parser SHALL produce `AndNode(ContainsNode("("), ContainsNode(")"))`

### Requirement: TokenBuffer Abstraction

The `TokenBuffer` class SHALL provide `peekToken`, `consume`, `isAtEnd`, and `isNotAtEnd` operations over a `[Token]` array, mirroring the `TokenCharacterBuffer` interface at the token level. It SHALL support safe subscript access returning `nil` for out-of-bounds indices.

#### Scenario: Safe out-of-bounds access

- **GIVEN** a `TokenBuffer` with all tokens consumed
- **WHEN** `peekToken()` is called
- **THEN** it SHALL return `nil` rather than crashing

### Requirement: Expression Tree Nodes

The parsing module SHALL produce an expression tree composed of: `AnythingNode` (wildcard), `ContainsNode` (leaf term with both `String` and precomputed `CString` representations), `NotNode` (unary negation), `AndNode` (binary conjunction), and `OrNode` (binary disjunction). All nodes SHALL conform to the `Expression` protocol.

#### Scenario: Expression evaluation via StringExpressionSatisfiable

- **GIVEN** an expression tree and a value conforming to `StringExpressionSatisfiable`
- **WHEN** `isSatisfied(by:)` is called
- **THEN** the tree SHALL recursively evaluate using `contains(phrase:)` on leaf nodes, boolean AND/OR on branch nodes, and negation on `NotNode`

#### Scenario: Expression evaluation via CStringExpressionSatisfiable

- **GIVEN** an expression tree and a value conforming to `CStringExpressionSatisfiable`
- **WHEN** `isSatisfied(by:)` is called
- **THEN** the tree SHALL evaluate using `matches(needle:)` with precomputed `CString` values for performance

### Requirement: Negation Normal Form

The `ContainmentEvaluator` SHALL transform an expression tree into negation normal form by pushing `NotNode` through `AndNode` and `OrNode` using De Morgan's laws. This normalization SHALL enable phrase extraction from the tree.

#### Scenario: De Morgan's law application

- **GIVEN** a `NotNode` wrapping an `AndNode(A, B)`
- **WHEN** `normalizedEvaluable()` is called
- **THEN** the evaluator SHALL produce `OrNode(NotNode(A), NotNode(B))`

#### Scenario: Recursion depth limit

- **GIVEN** a deeply nested expression tree
- **WHEN** normalization exceeds `maxRecursion` (default 50)
- **THEN** the evaluator SHALL throw `RecursionTooDeepError`

### Requirement: Phrase Extraction

The `PhraseCollectionConvertible` protocol SHALL extract candidate phrases from a normalized expression tree. `ContainsNode` SHALL return its string, `NotNode` SHALL return an empty array (since negated terms are excluded), `AnythingNode` SHALL return an empty array, and `AndNode`/`OrNode` SHALL concatenate phrases from both children.

#### Scenario: Phrase extraction from search query

- **GIVEN** the expression tree for `"foo AND bar AND NOT baz"`
- **WHEN** `ContainmentEvaluator.phrases()` is called
- **THEN** the result SHALL be `["foo", "bar"]`, excluding the negated term

### Requirement: Module Dependency Flow

The dependency flow SHALL be unidirectional: the Tokenization module SHALL have no dependencies on other modules. The Parsing module SHALL depend on token types from the Tokenization module. The NormalForm module SHALL depend on expression types from the Parsing module. The `Either` type SHALL be shared infrastructure used by the Tokenization module. The top-level `SearchExpressionParser.swift` SHALL depend on both Tokenization and Parsing modules.

#### Scenario: Tokenization independence

- **GIVEN** the Tokenization module
- **WHEN** its imports and type references are inspected
- **THEN** it SHALL not reference any types from the Parsing or NormalForm modules

## Technical Notes

- **Implementation**: `Sources/SearchExpressionParser/` with subdirectories `Tokenization/`, `Parsing/`, `NormalForm/`
- **Dependencies**: No external dependencies; single-target Swift package (swift-tools-version 5.5, macOS 10.13+)
- **Heritage**: `TokenCharacterBuffer` and `Either` originate from DDMathParser (Dave DeLong, MIT License)
- **Dual evaluation paths**: `Expression` supports both `StringExpressionSatisfiable` (Swift string matching) and `CStringExpressionSatisfiable` (C-level `strstr` matching for performance)
- **Error types**: `TokenizerError` (tokenization failures with character ranges) and `ParseError` (parsing failures as an enum)
